# Product

## Register

product

## Users

opnr.gg is for TFT players who want more control over how they find, compare, sort, and filter builds than any single scraped source gives them. They are usually in a fast decision-making context: scouting comps, checking item and augment signals, comparing phases, or deciding whether a build path is worth pursuing.

## Product Purpose

opnr.gg aggregates TFT build data from multiple sources into a desktop-first interface that gives players more agency over the build discovery workflow. The product exists to turn scattered comp signals into a controllable workspace for scanning builds, filtering sources and phases, inspecting boards, reading levelling guidance, comparing similarities, and making faster decisions with more confidence.

Success means the interface feels more powerful than the source sites it combines. Players should be able to shape the view around their own questions instead of adapting to one service's fixed ranking, filtering, or presentation model.

## Brand Personality

Cool, edgy, technical.

The product should feel like a dense player-controlled TFT build cockpit: sharp, fast, inspectable, and a little dangerous. The current language leans into a dark dot-matrix field, bracketed controls, panel chrome, compressed metadata, Rajdhani display type, neon signal color, and an arcade-terminal energy. It should feel engineered for high-control play, not styled as a generic analytics dashboard.

## Anti-references

Do not make opnr.gg feel like a serif editorial publication, soft SaaS marketing page, airy landing-page hero, generic card dashboard, or decorative glassmorphism showcase. Avoid cozy productivity-app softness, corporate BI polish, and designs that prioritize explanation over command.

The interface should not look like a blog, article, brochure, startup homepage, or ordinary TFT guide site. It should not bury source data inside vague summaries when the player needs control, comparison, and inspection.

## Design Principles

1. Player control first: sorting, filtering, panel placement, and inspection should feel like the user's tools, not a preset view they tolerate.
2. Dense but legible: compact information is a feature, but hierarchy, spacing, and labels must make the next useful action obvious.
3. Data stays inspectable: source, phase, evidence, items, augments, board state, and guide notes should remain close enough to compare without losing context.
4. Technical style serves speed: edgy terminal-like styling is welcome when it increases focus, affordance, and confidence.
5. Performance is part of the design: background effects, dragging, resizing, and hover inspection must stay responsive because the interface is meant to be actively manipulated.

## Accessibility & Inclusion

Aim for WCAG AA where practical. Preserve keyboard access for core controls, visible focus states, and text contrast that survives the dark technical visual direction. Support reduced-motion preferences for nonessential animation, and avoid relying on color alone for ranks, states, selections, or warnings.
